# 实现清单 v3：开场分流 + 双路径 + 收敛封顶

> **最高约束：不新建任何模块、不新建任何端点、不引入多模型/Coordinator。**
> 全部改动落在 `prompts/consultant.js`、`server.js`、`public/` 三处。

---

## 0. 病根与总思路

真实测试暴露：用户答「商业验证阶段，没有客户」，AI 却问「是什么导致了没有客户」——对早期项目问组织因果链是荒谬的。

**结论：缺一个开场分流。** 两条路径：
- **early** — 早期/无客户/想法阶段 → 验证式轻流程（4-5轮）
- **org** — 有真实运营/组织问题 → 世界模型 6-Stage

两条路都：不诊断、不给答案、最后产出同构的 7 天实验卡。

---

## 1. 开场隐性分流

**铁律：绝不问客户「你是早期还是成熟企业」。** AI 从开场 1-2 轮隐性判断。

| 信号 | 判为 | path |
|------|------|------|
| 没有客户 / 还没上线 / 刚开始 / 想法阶段 / 没验证 / 没收入 | 早期 | `early` |
| 客户在流失 / 利润下滑 / 团队问题 / 已在运营 / 有真实数据 | 组织 | `org` |
| 模糊不清 | 再问一句（最多1轮），仍不清按 `early` 兜底 |

**回归测试**：输入「商业验证阶段，没有客户」必须判 `early`。

---

## 2. 早期路径（path=early，4-5轮）

目标：帮早期创业者走到「7天需求验证实验」。

- **E1 押预测**：「找5个目标客户，你觉得几个愿意付钱？给个数字。」
- **E2 撬「先造后验」**：「在产品做完前，你能拿什么最小的东西验证？」
- **E3 定义「验证成功」**：「什么信号算验证成功？给个数字+时间。」
- **E4 最小实验**：「未来7天，你能做的最小验证是什么？」

**封顶**：≤5轮，到顶出实验卡。
**禁止**：给商业建议、替他设计实验。

---

## 3. 组织路径（path=org，6-Stage）——撞击式提问

沿用 6-Stage，但 **Stage 3/4 必须改撞击式**：

**禁止（空问法）：**
- ❌「还有什么可能在影响？」
- ❌「如果解释不完整，可能缺了什么？」

**要求（撞击式，具体、一口能答）：**
- ✅「这段时间，新客户数量是涨还是跌？」
- ✅「老客户流失时，每单平均金额变了吗？」

从案例库取缺失变量方向，翻译成针对客户情境的、可用「涨/跌/数字/有没有」回答的问题。

---

## 4. 收敛封顶（两条路共用）

三层硬封顶：

1. **每 Stage 提问硬上限**：org 路 Stage 1-4 各 ≤2-3 轮，到顶无条件推进
2. **客户选择结束阶段**：Stage 3 撞 2-3 个问题后问「哪个最让你意外？」客户一选就结束
3. **全场总轮数**：硬上限 ~15 轮，到顶强制进 Stage 5+6 收尾

---

## 4.5 问题降级

三级难度：

| Level | 题型 | 示例 |
|-------|------|------|
| L1 | 开放式 | 「在你看来，是什么导致了这个？」 |
| L2 | 填空式 | 「你感觉是 ___ 影响了 ___。」 |
| L3 | 选择式 | 「A) 最近一月 B) 一季度 C) 一年 D) 更早」 |

**🔴 L3 红线（必须写死）：**
- **只有事实题**（时间/数量/涨跌/有没有）可降到 L3
- **归因题/定义题**最多降到 L2，**绝不给选项**
- AI 输出 `question_kind: "fact|attribution|definition"`，server 校验：`question_kind != "fact"` 时强制清空 options

**降级规则**：
- 连续2次模糊 / 连续2次过短 / 1模糊+1过短 → 降一级
- 连续2次好回复 → 升一级
- 切换 path/stage 时重置 L1

## 5. 状态结构（server.js）

```javascript
state = {
  path: "unknown|early|org",   // 开场分流，确定后锁定
  stage: 1,                    // org: 1-6, early: 1-4
  stage_turn: 0,               // 当前阶段已提问轮数
  total_turns: 0,              // 全场轮数
  difficulty: "L1",            // L1|L2|L3
  vague_streak: 0              // 连续模糊计数
}
```

输出 JSON：
```javascript
{
  reply: "...",
  path: "early|org",
  stage: 1,
  stage_turn: 1,
  total_turns: 1,
  causal_chain: [],            // 仅 org Stage 2
  difficulty: "L1",
  question_kind: "fact",       // fact|attribution|definition
  options: [],                 // 仅 L3 + fact 时非空
  session_hint: null,          // null|"approaching_end"
  internal_note: "...",        // 前端不可见
  session_complete: false
}
```

server 返回前：
1. 剥掉 `internal_note`
2. `question_kind != "fact"` 时强制 `options = []`

## 6. 前端改动（public/）

1. **删除进度条**（「故事→因果→发现→好奇→重定义→实验」）
2. **单页自然对话**

## 7. 输出卡（两条路同构）

| org 路 | early 路 |
|--------|----------|
| 当前问题定义 | 当前想法/挑战 |
| 当前世界模型（因果链） | 核心假设 |
| 隐藏假设 | 被撬动的假设 |
| 可能缺失的变量 | 预测 vs 待验证 |
| 好奇问题 | 验证成功定义 |
| 更新后的问题定义 | 更新后的问题定义 |

## 9. 验收标准

1. **回归测试**：「商业验证阶段，没有客户」→ 必须走 early 路
2. **收敛测试**：early ≤5轮，org ≤15轮
3. **L3红线测试**：事实题可出L3选项；归因/定义题即使连续模糊也绝不出options
4. **真人验证**：≥半数触发「我从没这样想过」+ 问题定义由客户自述
